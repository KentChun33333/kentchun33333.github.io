Nowadays, we have jira and confluence to help us managing the task and actual works. However, without good strategy, it can not help much.

Task Type with Expected Outputs

It is important to have clear definition on the expected outputs of each type of tasks.

Task Type Description Expected Outputs
Issue New issue, query’s EmailsError MessageProblem Statement
Request New Functions / Idea from others Problem Statement | Measurement on Success
Idea anything Problem Statement | Potential solutions | Documentation
Solutioning - Idea propose- Communications - Designs Emails and Documentation base on Request DocumentationResult can be decomposed of multiple problem statement. each with different solutions
Implementation - Feature Functions need to be implement in - code- document- …etc PR based on Solutioning DocumentationPass BuildBot … etc
Runs - Run Experiment from feature branch - Runs from Dev branch for UAT- Runs from Master for Production (IT)- chase IT/DE …etc on Data Issues- … etc Experiment DocumentationReproducibility, that config and metric is on Database, so ppl can check into git-version and reproduce the result.
Output for UAT - Feature Dictionary- Data Dictionary - Reports  ..etc- Model Documentation
Get Feedback of UAT DocumentationUser Confirmation Email
Implement to Dev PR + Documentation of UAT result +Documentation of Model Government + … etc

Documentations

Document Type Description Comment
Product Dictionary The name and description base on a set of info. - Model Feature dictionary: for feature monitoring, user communication, explanation- Input data dictionary : to database management, communication to DE, monitoring data quality- Output data dictionary : to monitor model performance- All I/O Table dictionary : to monitor in and out in table-lvl- Model dictionary : for model selection, experiment explanation. - Training Method dictionary- Metric Dictionary- Agent Dictionary- Experiment Dictionary
Project Dictionary - Task Tracking Table | and its dictionary- Issue summary Table | and its dictionary- Task Type Dictionary - …
UML The abstraction of code. usually we use it in solutioning phase to penetrate the ideas to target audience like develops/users/advocators …etc - Class Diagram- User Story- …
Project LOG The project catalog with timestamp. This can be generated by Task Management Table record with time-stamp for auditing trails, the content point to the relevant pages.
Experiment Result currently in a more loosen definition. need to clarify with the 1. problem statement / objectives2. current code version, input versions … etc 3. experiment config, result tables…etc
User confirmation records upload the importance evidence Report tracking Email Tracking
Success Metrics the key measurement of project/products during a period of time - UAT checklist- Model Performance- Forward testing Tracking- System Performance- QPS Peak-Measurement- API testing | J-meter and glucos/postman…etc
Evaluation Document document how we validate project/ produce/ model ..etc
